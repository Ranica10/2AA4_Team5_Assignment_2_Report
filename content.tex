\section{Executive summary}

The second assignment of this course has enhanced our previous implementation of the Catan simulator to accommodate **human gameplay**. New features of this enhanced version are the implementation of a **command-line interface** for human players to enter their commands, the implementation of the **robber** function, which is activated when the rolled numbers are 7, and the implementation of **JSON export** for visualization of the game. This version of the simulator has been enhanced to **pause** between human turns, waiting for the human to enter "Go" to continue the game. A total of **18 unit tests** were written before this implementation, ensuring the detection of bugs early in the process. This version of the simulator has been successfully enhanced to accommodate human gameplay while retaining all original functionality.

\section{Changing and extending your design}

\textbf{What were the main structural changes in your original design? Was it easy to localize the change and assess its impact compared to directly changing the code (without messing around with the design)?}\newline\newline
During A2, several structural changes were made to improve the design of the Catan implementation and support new requirements. One of the biggest changes was how players take turns. In A1, the game only supported a computer player, and turns were executed randomly. The logic for taking a turn was handled directly inside the Board class, and all actions, such as building roads, settlements, or cities, were randomly selected. \newline
\newline
For A2, we needed to support both human and computer players. To accommodate this change, we redesigned the player structure. The Player class was made abstract, and an abstract method takeTurn() was made. Two subclasses were created:

\begin{itemize}
    \item ComputerPlayer: overrides takeTurn() and randomly chooses actions (similar to A1)
    \item HumanPlayer: overrides takeTurn() and allows the user to input commands using Scanner
\end{itemize}
This design allows different player types to share common functionality while implementing their own turn behavior. \newline
\newline
Another important structural addition was the TurnManager class. This class manages the entire turn lifecycle, including actions during that turn, and determines who is taking the turn- human or computer. Instead of duplicating turn logic for each player type, the TurnManager simply calls the appropriate player's takeTurn() method. The TurnManager also manages the turn state using a TurnState enum, while players interpret commands using a UserInput enum (such as ROLL, LIST, BUILD, and GO). \newline
\newline
We also added a new Robber class to represent the robber piece in the game. The robber is triggered when certain conditions occur (such as a dice roll of 7). The robber logic needed to interact with both the Board and the player turn flow, which required modifications in the turnManager implementation and in the Board class. This was one of the more difficult features to implement because it involved coordinating multiple classes and game states. \newline
Several other improvements were made based on feedback from A1: 

\begin{itemize}
    \item The road was removed from extending the building, since a road is not technically considered a building. 
    \item New methods were added to the Player class, such as listResources() and hasResources().
    \item The takeTurn() method was removed from the Board class to improve separation of responsibilities.
    \item Some class names were corrected for clarity and consistency (Roads to Road, Cities to City). 
\end{itemize}
Some changes were easier to localize than others. The introduction of HumanPlayer and ComputerPlayer was relatively straightforward because both classes extend the Player class and override the same abstract method. This allowed us to add new player behavior without heavily modifying the rest of the system. However, other changes were more complex. Implementing the TurnManager state machine and the Robber feature required modifying multiple parts of the system because they interact with the board, players, and turn logic. Managing both the UserInput enum and the TurnState enum also required careful coordination.  \newline
\newline
Overall, making these changes at the design level first helped us better understand the impact of the modifications compared to directly changing code without considering the design. The use of abstraction and separate classes helped isolate many of the changes and improved the overall structure of the program. \newline

\textbf{Why was the automaton-based modelling useful and what benefits does it provide compared to the specification in the attached PDF? How did you translate the automaton-based model to program code?}

Automaton-based modelling was useful because it allowed us to clearly represent the different states of a player’s turn and the transitions between them. In a game like Catan, a turn follows a specific sequence of actions, such as rolling dice, performing actions like building or trading, and then ending the turn. Using an automaton model helped us visualize this flow as states and transitions, which made the logic easier to understand and organize. Compared to the specification in the PDF, which mainly describes the rules in text form, the automaton model provides a visual and structured representation of the game flow. This makes it easier to see which actions are valid at each stage of the turn and what conditions cause the game to move from one state to another. It also helps avoid illegal actions, since the program only allows transitions that are defined in the model. \newline
\newline
Another benefit is that automaton-based modelling improves maintainability and debugging. Because each state has clearly defined transitions, it becomes easier to identify where errors occur and to modify the system if new rules are added. \newline
\newline
We translated the automaton model into code by implementing a state machine in the TurnManager class. Each state of a player's turn is represented using the TurnState enum, which includes STARTTURN, ROBBER, DOACTION, and ENDTURN. \newline
\newline
The TurnManager keeps track of the current state and changes it based on game events and player actions. For example, a turn begins at STARTTURN, where the dice are rolled. If a 7 is rolled, the game moves to the ROBBER state to handle the robber logic. After that, the player enters DOACTION, where they can perform actions such as building or listing resources. Finally, the game moves to ENDTURN, where the turn finishes and control passes to the next player. \newline
\newline
Player commands are represented using a UserInput enum, and the TurnManager checks both the current state and the command to determine what actions are allowed and how the state should transition. This directly implements the automaton model by mapping states to enums and transitions to logic in the TurnManager.\newline

\section{Revise the program code and finish the functionality}

\textbf{Which parts of your system did break and how did you resolve those issues?}
\newline \newline
Several parts of the system broke when we improved on our original implementation. 
\newline\newline
One issue occurred when the executing turns was moved from Board class to TurnManager and Player classes. Originally, the Board class handled the turn logic directly. After added the TurnManager and the state machine, some methods that needed longer worked correctly because the flow had changed. This caused issues where turns were not progressing properly or actions were being triggered at the wrong time. We resolved this by putting all turn-related logic inside TurnManager and ensuring that Board was only responsible for managing the board state, for example placing a building and resource production.
\newline\newline
We also had issues when adding the Robber functionality. The robber works with both the dice roll and the board state, which meant many parts needed to work together to run correctly. This was solved by adding a robber management class and calling it from the TurnManager when the game state was reached.
\newline\newline
Overall, most of the issues were related to refactoring responsibilities between classes. These problems were resolved by separating responsibilities between Board which contained the board state, Player which contained the player behavior, and TurnManager which contained the turn flow.
\newline\newline

\textbf{What was the benefit of having unit tests at this stage?}
\newline\newline
Having unit tests at this stage was beneficial because our system underwent significant structural changes when introducing new components such as the TurnManager, command parsing with regex, and different types of players. These changes required refactoring existing classes and modifying how different components interacted. Without tests, it would have been difficult to verify whether the new changes unintentionally broke the functionality that already existed.
\newline \newline
The unit tests provided an extra layer of safety by quickly identifying errors introduced during the refactoring stage. For example, when modifying the turn execution logic and converting that method into a state machine, the tests ensured that core game behaviors such as building structures, resource updates, and player scoring continued to function correctly. This allowed changes to be made without needing to worry about the previous features because any unexpected behavior could be detected immediately by running the tests.

\textbf{Elaborate on the OO mechanisms and SOLID principles in your implementation. Are these any different from the ones used in the design? If they are: why?}
\newline \newline
In the revised implementation, object-oriented mechanisms were used to improve the structure of the program as new functionality such as human input and turn management was introduced. The most important mechanism applied was encapsulation, where each class is responsible for managing its own state and behavior. For example, now the Board class only encapsulates the state of the game board including nodes, tiles, and building placement without focusing too much on the details of taking turns (as it did previously). 
\newline \newline
Instead, the TurnManager class was assigned the responsibility to encapsulate all the logic for managing a player’s turn, including the state transitions between rolling the dice, performing actions, and ending the turn. This directly ties into the SOLID principle of single-responsibility between classes. Instead of placing all gameplay logic inside the Game or Board classes, the introduction of the TurnManager allows turn-related logic to be handled independently. The Game class is now responsible only for managing the overall game flow and iterating through players and rounds, while the TurnManager handles the sequence of actions within a turn.
\newline \newline
In addition, the Open/Closed Principle was used by designing the system so that additional commands or actions can be added without needing to conduct any major refactoring/structural changes. For example, the TurnManager processes commands through a switch structure based on a pre-defined enum. This means that new commands could be added with minimal modifications while preserving the overall structure of the system by simply adding it to the Actions enum and then including a new case in the switch statement.
\newline \newline
Compared to the original design, the main difference is the introduction of the TurnManager as a separate control component. As previously mentioned, in the earlier design, some of the turn-related logic was more tightly coupled with other classes such as Board, which previously handled the logic for doing the player’s turn. Overall, the OO principles used in the design stage remained largely the same, but the new implementation/features required a few adjustments to ensure that responsibilities were clearly distributed across classes while supporting the new functionality.
